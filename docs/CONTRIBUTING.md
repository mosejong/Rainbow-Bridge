# 🤝 협업 규칙 (CONTRIBUTING)

> 우리는 현업처럼 일합니다. 이 문서는 **팀원 전원 필독**입니다.
> Git 명령어 자체가 처음이라면 먼저 [GIT_GUIDE.md](GIT_GUIDE.md) 부터 읽으세요.

---

## 0. 한 줄 요약

> **`main`·`dev` 에 직접 push 금지. 내 이름 브랜치 → PR → 리뷰 → `dev` 에 merge.**
> (`main` 은 MVP 완성·서비스 가능 시점에만 `dev` 에서 승격)

---

## 1. 브랜치 전략 (인당 1브랜치 + dev 통합)

복잡한 전략 대신, 초보팀에 가장 단순한 **"인당 1브랜치"** 를 씁니다.
**각자 자기 이름 브랜치 1개**에서만 작업하고, 작업이 끝나면 PR로 **`dev`** 에 합칩니다.
`main` 은 **MVP가 완성되어 서비스 가능할 때**만 `dev` 에서 끌어올립니다.

```
main ──────────────────────────────●─────▶   (MVP/배포 시에만 dev→main PR)
dev  ──●──────●──────●──────●──────▶          (평소 통합 브랜치, PR로만 변경)
        \     /\     /
      mosejong  kimyunhan ...                 (각자 본인 브랜치 → dev로 PR)
```

- `main` : **서비스 가능한 안정 버전**. 평소엔 안 건드림. MVP 완성 시 `dev → main` PR로만 승격.
- `dev` : **평소 작업이 모이는 통합 브랜치.** 직접 push ❌, **PR 머지로만** 변경.
- **내 이름 브랜치** : 내가 작업하는 곳. 한 번 만들면 **계속 사용**(머지 후에도 삭제 ❌).

> ⚠️ `main` 과 `dev` **둘 다 직접 push 금지.** 항상 내 브랜치 → PR → `dev`.

### 내 브랜치 이름 (영문)

| 이름 | 브랜치 | 이름 | 브랜치 |
|------|--------|------|--------|
| 모세종 | `mosejong` | 정환주 | `junghwanju` |
| 김윤한 | `kimyunhan` | 민경이 | `mingyeongi` |
| 반소람 | `bansoram` | 장민수 | `jangminsu` |

> 한 사람 = 한 브랜치 = 본인 영문 이름. 헷갈리지 않게 평생(?) 이 브랜치만 씁니다.
> 작업 단위(기능 하나 끝날 때마다)로 PR을 올려 main에 합치고, 같은 브랜치에서 다음 작업을 이어갑니다.

## 2. 커밋 메시지 컨벤션

**Conventional Commits** 형식을 따릅니다.

```
<타입>: <한글로 무엇을 했는지 한 줄>

(필요하면 본문에 왜/어떻게 추가)
```

### 타입

| 타입 | 사용 시점 | 예시 |
|------|-----------|------|
| `feat` | 새 기능 | `feat: 반려동물 프로필 등록 API 추가` |
| `fix` | 버그 수정 | `fix: 감정 체크인 날짜 저장 오류 수정` |
| `docs` | 문서 | `docs: README 빠른시작 보강` |
| `style` | 포맷/세미콜론 등 (로직 변화 없음) | `style: black 포맷 적용` |
| `refactor` | 리팩터링 | `refactor: 메시지 생성 로직 service로 분리` |
| `test` | 테스트 | `test: 프로필 API 테스트 추가` |
| `chore` | 빌드/설정/패키지 | `chore: requirements.txt 정리` |

### 좋은 예 / 나쁜 예

```
✅ feat: 위험 감정 감지 시 1393 안내 응답 추가
✅ fix: TTS 음성 톤 파라미터 누락 수정

❌ 수정함
❌ ㅁㄴㅇㄹ
❌ final final 진짜최종
```

> **한 커밋 = 한 가지 변경.** 여러 기능을 한 커밋에 섞지 마세요.

## 3. Pull Request (PR) 규칙

1. 작업(기능 하나) 끝나면 `내이름` 브랜치 → **`dev`** 로 PR 생성 (base가 `dev`인지 꼭 확인)
2. PR 제목도 커밋 컨벤션 따름: `feat: 추모 메시지 생성 API`
3. PR 본문에 **무엇을 / 왜 / 어떻게 테스트했는지** 작성 (템플릿 자동 제공됨)
4. **리뷰어 최소 1명 승인 후** 머지
5. 머지 방식: **Squash and merge** 권장 (커밋 히스토리 깔끔)
6. ⚠️ **내 브랜치는 삭제하지 않습니다.** 머지 후 `git checkout 내이름 && git merge dev` 로 dev 최신을 내 브랜치에 받아오고, 같은 브랜치에서 다음 작업 계속.

> 🚀 **main 승격:** MVP가 완성되어 시연/배포 가능한 시점에 **`dev → main` PR**을 따로 올립니다 (PM 모세종이 진행).

> 리뷰는 "트집"이 아니라 "같이 더 좋게"입니다. 모르면 PR에 댓글로 물어보세요.

## 4. 코드 리뷰 매너

- 24시간 안에 리뷰 (안 되면 멘션으로 알리기)
- 칭찬도 코멘트로 남기기 👍
- "이거 왜 이렇게 했어요?" 대신 "여기 이렇게 하면 어떨까요?"
- 사소한 제안은 `nit:` 접두사 (예: `nit: 변수명 더 명확하게`)

## 5. 코드 스타일

| 영역 | 도구 / 규칙 |
|------|-------------|
| Python | `black` (포맷), `ruff` (린트), 함수/변수 snake_case |
| 폴더/모듈 | 소문자 + 언더스코어 |
| API 경로 | `/api/v1/...` 소문자, 복수형 명사 (`/pets`, `/emotions`) |
| 환경변수 | 코드에 하드코딩 ❌ → 반드시 `.env` |

> 비밀키·API키는 **절대** 커밋 금지. `.env` 에만. 실수로 올렸다면 즉시 팀에 알리고 키 재발급.

### 🤖 자동 검사 (CI)

`backend/` 코드가 바뀐 PR은 GitHub Actions가 자동으로 `ruff`·`black`·`pytest` 를 돌립니다.
빨간불을 피하려면 **push 전에 로컬에서 한 번** 정리하세요:

```bash
cd backend
ruff check . --fix     # 린트 자동 수정
black .                # 포맷 자동 정리
pytest -q              # 테스트 (있으면)
```

> 지금은 "권장" 단계라 빨간불이어도 머지는 됩니다. 테스트가 쌓이면 통과 필수로 승격합니다.
