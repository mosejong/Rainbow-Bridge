## 1. 브랜치 전략 (GitHub Flow)

복잡한 git-flow 대신, 우리 규모에 맞는 **GitHub Flow** 를 씁니다.

```
main  ────●────────●────────●──────▶   (항상 동작하는 코드만)
           \        \        /
            \        feature/llm-message  (PR로 합침)
             feature/profile-api
```

- `main` : **항상 배포 가능한 상태**. 직접 커밋 ❌. PR 머지로만 변경.
- `feature/*` : 기능 개발 브랜치. 여기서 작업하고 PR 올림.
- `fix/*` : 버그 수정 브랜치.
- `docs/*` : 문서 작업 브랜치.

> 우리는 짧은 스프린트(약 3주)라 `develop` 브랜치는 두지 않습니다. main + feature 만으로 갑니다.

### 브랜치 이름 규칙

```
<타입>/<영역>-<짧은설명>

feature/profile-api          # 프로필 입력 API
feature/llm-memorial-message # 추모 메시지 생성
fix/timeline-sort-bug        # 타임라인 정렬 버그
docs/setup-guide             # 셋업 문서
```

- 소문자 + 하이픈(`-`) 사용. 한글/공백 ❌
- `<영역>` 은 자기 담당 영역 (profile, emotion, llm, tts, timeline, video, eval ...)

## 2. 커밋 메시지 컨벤션

**Conventional Commits** 형식을 따릅니다.

```
<타입>: <한글로 무엇을 했는지 한 줄>

(필요하면 본문에 왜/어떻게 추가)
```

### 타입

| 타입 | 사용 시점 | 예시 |
|------|-----------|------|
| `feat` | 새 기능 | `feat: 반려동물 프로필 등록 API 추가` |
| `fix` | 버그 수정 | `fix: 감정 체크인 날짜 저장 오류 수정` |
| `docs` | 문서 | `docs: README 빠른시작 보강` |
| `style` | 포맷/세미콜론 등 (로직 변화 없음) | `style: black 포맷 적용` |
| `refactor` | 리팩터링 | `refactor: 메시지 생성 로직 service로 분리` |
| `test` | 테스트 | `test: 프로필 API 테스트 추가` |
| `chore` | 빌드/설정/패키지 | `chore: requirements.txt 정리` |

### 좋은 예 / 나쁜 예

```
✅ feat: 위험 감정 감지 시 1393 안내 응답 추가
✅ fix: TTS 음성 톤 파라미터 누락 수정

❌ 수정함
❌ ㅁㄴㅇㄹ
❌ final final 진짜최종
```

> **한 커밋 = 한 가지 변경.** 여러 기능을 한 커밋에 섞지 마세요.

## 3. Pull Request (PR) 규칙

1. 작업 끝나면 `feature/*` → `main` 으로 PR 생성
2. PR 제목도 커밋 컨벤션 따름: `feat: 추모 메시지 생성 API`
3. PR 본문에 **무엇을 / 왜 / 어떻게 테스트했는지** 작성 (템플릿 자동 제공됨)
4. **리뷰어 최소 1명 승인 후** 머지
5. 머지 방식: **Squash and merge** 권장 (커밋 히스토리 깔끔)
6. 머지 후 브랜치 삭제

> 리뷰는 "트집"이 아니라 "같이 더 좋게"입니다. 모르면 PR에 댓글로 물어보세요.

## 4. 코드 리뷰 매너

- 24시간 안에 리뷰 (안 되면 멘션으로 알리기)
- 칭찬도 코멘트로 남기기 👍
- "이거 왜 이렇게 했어요?" 대신 "여기 이렇게 하면 어떨까요?"
- 사소한 제안은 `nit:` 접두사 (예: `nit: 변수명 더 명확하게`)

---

## 5. 코드 스타일

| 영역 | 도구 / 규칙 |
|------|-------------|
| Python | `black` (포맷), `ruff` (린트), 함수/변수 snake_case |
| 폴더/모듈 | 소문자 + 언더스코어 |
| API 경로 | `/api/v1/...` 소문자, 복수형 명사 (`/pets`, `/emotions`) |
| 환경변수 | 코드에 하드코딩 ❌ → 반드시 `.env` |

> 비밀키·API키는 **절대** 커밋 금지. `.env` 에만. 실수로 올렸다면 즉시 팀에 알리고 키 재발급.

## 6. 이슈 / 작업 관리

- 할 일은 GitHub Issues 로 관리 (또는 [PROGRESS.md](PROGRESS.md))
- 이슈 → 브랜치 → PR 흐름
- PR 본문에 `closes #이슈번호` 쓰면 머지 시 이슈 자동 종료
