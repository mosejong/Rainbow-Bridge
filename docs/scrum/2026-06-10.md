## 주요 논의

### 1. 앱 테스트 영상 / iOS 문제

- 강사님이 **시연 영상** 요청 → 팀이 찍어둔 영상 있음
- 아이폰에서 Expo Go SDK 54 버전 다운이 안 되는 문제 → iOS 테스트 불가 상태 보고

### 2. 서비스 진입점 재확인

- 강사님 재확인: **"화면은 죽고 나서 가입하는 게 아니라, 시한부 판정 받고 나서 가입하는 것"**
- 즉, 이별 이후가 아닌 **생전(시한부 선고 시점)** 진입이 맞는 흐름임을 재강조

### 3. 회의록의 목적

- 회의록 안 쓰면 의사소통 안 된 것으로 간주
- **팀장(모세종)이** 강사님 미팅 녹음 → 안건들 보고 회의 추가 진행 → 회의록 작성 → 팀 전원 배포 → 추가 회의 → 결정 → 팀원들 확인

### 4. 데이터 수집 로직

- 영상 재생 로그 수집 로직 팀이 짜고 있음 (장민수 담당)
- 강사님: **"얼마나 봤는지(시청 비율)까지 수집"** — 재생 시작만이 아니라 실제 시청 길이가 중요
- 데이터 동기화 방식 → **하루 1회 배치로 확정** (2026-06-10)

### 5. 데이터 기반 기능 구현

- 강사님: **"데이터 분석가가 어려운 이유는 데이터를 직접 만들어야 하기 때문"**
- 실 데이터 수집 없이 분석 불가 → 수집 구조부터 설계 필요
- **"데이터를 뽑아서 근거를 바탕으로 기능을 구현해보자"** — 단순 구현이 아닌 데이터 근거 있는 기능
- 강사님: **"한 번 보면 끝까지 볼 컨텐츠"** → 시청 비율 측정 불필요, **영상 보기 버튼 클릭 시점**을 play_count로 카운트하는 것으로 확정
- 데이터 사이언티스트 역할: 임무 분장을 잘하는 것 + 일반 실 데이터가 핵심
- **수집 → 분석 → 근거 파이프라인 (강사님 의도 구체화)**
  ```
  영상 보기 클릭(play_count) → 하루 1회 배치 집계 → ⑧ 리포트 회복 분석
                                                    → "재생 감소 = 일상복귀" 가설 검증
  ```
  → 강사님이 강조한 "데이터를 뽑아서 근거로 기능 구현"의 실제 파이프라인
