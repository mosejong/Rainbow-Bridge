## 슬라이드 2 — 문제: 펫로스, 얼마나 심각한가 *(3분)*

**말할 내용:**

> "오늘 한국에서 반려동물을 키우는 인구는 약 1,546만 명, 전체 가구의 29.2%입니다.
> 이 중 매년 수백만 마리가 세상을 떠납니다.
>
> 반려동물을 잃은 보호자의 애도는 평균 약 **10개월**, 보통 6개월에서 1년까지 지속됩니다.
> 1년이 지나도 약 **20%**가 애도 증상을 보고합니다 (Hughes & Lewis Harkin, 2022).
>
> 영국 대표 표본 연구에서 반려동물 사별자 중 **7.5%**가 prolonged grief disorder,
> 즉 1년 이상 지속되는 병적 애도 기준을 충족했습니다 (Hyland et al., 2026).
> 국내 연구에서도 펫로스 경험자의 **32~55%**가 복잡 애도·우울·불안·불면 증상을 보고했습니다 (Ahn et al., 2023).
>
> 그런데 우리 사회는 이 고통을 아직도 '별거 아닌 일'로 봅니다.
> '개 한 마리 죽었는데 왜 이렇게까지 슬프냐'는 말을 듣는 보호자가 많습니다.
>
> 더 중요한 건 시점입니다.
> 보호자가 가장 막막한 순간은 진단 직후, 수의사실을 나오는 그 순간입니다.
> 그 순간 곁에 있어줄 도구가 국내에 없습니다."

## 슬라이드 3 — 서비스 소개 + B2B2C 모델 *(3분)*

**말할 내용:**

> "레인보우 브릿지는 반려동물 **시한부 선고를 받은 순간부터 시작해**,
> 이별을 함께 준비하고, 이별 후 일상을 되찾을 때까지 곁에 있는 AI 케어 서비스입니다.
>
> 핵심은 **연속성**입니다.
> 지금까지의 펫 앱은 건강 관리 또는 메모리얼, 둘 중 하나였습니다.
> 저희는 시한부 → 생전 기록 → 이별 → 추모 → 회복까지,
> 보호자가 겪는 전체 과정을 하나의 흐름으로 연결합니다."

> "수익 모델은 B2B2C입니다.
> 수의사가 시한부 선고 직후 보호자에게 앱을 추천합니다.
> 보호자는 **무료**로 사용하고, 병원과 장례업체가 구독 채널이자 지불자가 됩니다.
>
> 이 구조가 중요한 이유는 진입 시점입니다.
> 보호자가 가장 막막한 그 순간에 앱을 만납니다.
> 그 시점부터 같이 시작합니다."

## 슬라이드 4 — 핵심 흐름 5단계 *(2분)*

**말할 내용:**

> "서비스 흐름은 크게 다섯 단계입니다."

```
① 시한부 선고  →  앱 설치, 프로필 등록
② 생전 기록    →  버킷리스트 / 일기·추억 / 사진
③ 사후 추모    →  AI 추모 편지 + TTS 낭독 + 영상
④ 회복 미션    →  논문 기반 5카테고리 일상복귀 미션
⑤ 정량 분석   →  감정 체크인 추이 / 영상 재생횟수 / 접속빈도
```

> "이 흐름 전체가 하나의 앱 안에 있고, 각 단계는 API로 연결됩니다.
> 어떻게 만들었는지 구조부터 보여드리겠습니다."

## 슬라이드 5 — 아키텍처 흐름도 *(4분)*

> *(슬라이드에 system_architecture_final.svg + rainbow_bridge_pipeline.svg 삽입)*
> *(user_flow_fixed.svg 또는 아래 텍스트 다이어그램 병행)*

**말할 내용:**

> "전체 시스템은 크게 네 레이어입니다."

```
[ 사용자 (보호자) ]
       │  HTTPS
       ▼
[ React Native + Expo (프론트) ]          ← 민경이 담당
       │ /api/ 프록시
       ▼
[ Backend (FastAPI, NCP 실서버) ]          ← 모세종·김윤한
       │
   ┌───┴──────────────┐
   │                  │
[ MongoDB ]    [ Gemini API + Google TTS ] ← Gemini: 반소람·정환주
   │                  │
[ ChromaDB RAG ]   [ LivePortrait GPU ]   ← 반소람 / 장민수
```

> "사용자 여정은 이렇습니다."

```
로그인 (JWT·SQLite)
  → 펫 프로필 등록
  → 감정 체크인 + 위기 감지
      └→ risk 2+ : 1393 즉시 안내 (생성 중단)
  → 추모 메시지 생성 (Gemini + RAG few-shot)
  → TTS 낭독 (Qwen3 로컬 TTS, GPU 서버)
  → 추모 영상 (사진 → LivePortrait → FFmpeg 합치기)
  → 일상복귀 미션 추천
  → 리포트 / 타임라인
```

> "백엔드는 레이어 구조를 엄격하게 지킵니다.
> `api/`는 얇게, 비즈니스 로직은 `services/`로, 요청·응답은 `schemas/`(Pydantic)로.
> MongoDB가 서비스 데이터 전체를, SQLite는 인증만 담당합니다.
>
> 실서버는 NCP (101.79.19.87), HTTPS는 Let's Encrypt + DuckDNS.
> GitHub Actions가 `dev` 브랜치 머지 시 NCP 자동 배포를 처리합니다.
> GPU 서버는 정환주님 홈서버 (RTX 5060) → ngrok 터널 → NCP 백엔드 연결 구조입니다."

## 슬라이드 7 — 중간 산출물 ① RAG 설계 *(3분)*

> *(담당: 반소람)*

**말할 내용:**

> "저희가 RAG를 쓰는 곳은 코드 기준으로 **3곳**입니다.
> 발표 자료에 4곳으로 적힌 것도 있는데, 실제 동작하는 코드 기준으로 말씀드립니다."

| category | 기능 | corpus | 검색 방향 |
|----------|------|:------:|----------|
| `memorial` | 추모·위로 메시지 생성 | 45건 | 감정 공감·이별 슬픔 |
| `mission` | 일상복귀 미션 추천 | 5건 | 회복 단계별 행동 유도 |
| `funeral` | 장례 안내 Q&A | 13건 | 절차·방법 정보 제공 |
| **합계** | | **63건** | |

> "동작 방식은 이렇습니다.
> 사용자 입력 → 의미가 가까운 corpus 글 top-3 검색 → LLM 프롬프트에 **참고 예시**로 주입 → 생성.
> 반드시 `where={"category": ...}` 필터를 써서 영역이 섞이지 않게 합니다.

```python
collection.query(
    query_texts=[query],
    where={"category": "memorial"},
    n_results=3
)
```

> RAG를 쓰지 않는 곳도 명확합니다.
> **위기 감지**는 안전 시스템에 예측 불가능한 검색 결과를 넣지 않기 위해 의도적으로 제외했습니다.
> **수의사 처치 안내**는 의료 조언 대체 위험 때문에 고정 텍스트로 확정했습니다.
>
> 코퍼스 텍스트는 git 추적해서 팀이 같이 리뷰합니다. 벡터 DB 바이너리만 .gitignore입니다."

## 슬라이드 8 — 중간 산출물 ② 위기 감지 4등급 *(2분)*

> *(담당: 반소람·모세종)*

**말할 내용:**

> "감정 체크인에서 사용자 입력이 들어오면 위기 감지가 실행됩니다.
> **2단 레이어 구조**입니다."

```
사용자 입력
    │
    ▼
L0 규칙 레이어 (키워드·패턴 매칭)
    │
    ▼
L1 Gemini 분류 (JSON 강제 출력)
    │
    ▼
보수적 융합 → 애매하면 상위 등급
    │
    ▼
risk_level 0 / 1 / 2 / 3
```

| 등급 | 의미 | 처리 |
|------|------|------|
| 0 | 정상 | 정상 응답 |
| 1 | 우려 | 정상 응답 + 내부 기록 |
| 2 | 경고 | **1393 안내** 포함 응답 |
| 3 | 긴급 | **1393 우선 노출**, 메시지 생성 중단 |

> "이 시스템을 검증하기 위해 골든셋 39건을 만들었습니다.
> 이 앱 맥락에서 정상 대화 — '보고 싶다', '힘들다' — 를 위기로 잘못 분류하면 안 됩니다.
> 반대로 '나도 따라가고 싶다'처럼 자해 암시가 있으면 반드시 감지해야 합니다.
> 골든셋 39건 기준 **미탐 0건**을 달성했습니다."

## 슬라이드 9 — 중간 산출물 ③ 회복 게이트 *(3분)*

> *(담당: 모세종)*

**말할 내용:**

> "펫로스 초기에 반려동물의 목소리나 1인칭 편지를 바로 보여주면
> 집착과 현실 부정으로 이어질 수 있습니다.
> 회복이 어느 정도 된 시점에 제공해야 치유 효과가 생깁니다.
>
> 저희는 이것을 **회복 게이트**로 구현했습니다."

**회복 점수 산식:**
```
recovery_score =
    감정 평균 (최근 7회, 40%)
  + 미션 완료율 (35%)
  + 체크인 꾸준함 (최근 14일 비율, 25%)
```

| 점수 | 상태 | 제공 콘텐츠 |
|------|------|------------|
| 0 ~ 49 | 초기·위기 | 위로 메시지·미션·타임라인 |
| 50 ~ 79 | 회복 중 | + **찌라시 카드** (잠김 상태 노출) |
| 80+ | 회복 | **추모 편지 1회 무료 생성** + LivePortrait 영상 |

> "50점이 되면 '잠긴 카드'가 보입니다.
>
> ```
> ┌─────────────────────────────────────────┐
> │  🔒  OO이가 남긴 편지                    │
> │  회복도  ██████████░░░░  63점           │
> │          80점이 되면 열립니다            │
> │  천천히 괜찮아요 🐾                      │
> └─────────────────────────────────────────┘
> ```
>
> '회복하면 OO이의 편지를 받을 수 있다'는 게 회복 동기가 됩니다.
>
> 80점 달성 시 편지를 **1회 무료로 생성해 영구 저장**합니다.
> 이후 점수가 떨어져도 기존 편지는 계속 열람 가능합니다. 토큰 비용 없이 DB에서 조회합니다.
> 새 편지를 원하면 크레딧을 구매하는 구조 — 이게 수익 모델로도 연결됩니다.
>
> 중요한 안전장치 하나: **risk_level 2 이상이면 점수 계산 자체를 중단**합니다.
> 위기 상황에서 콘텐츠 언락은 없습니다. 1393 안내가 우선입니다.
>
> 편지를 열람한 뒤 7일 동안은 집중 모니터링합니다.
> 감정 점수가 -20점 이상 하락하면 위로 메시지를 강화하고 미션 강도를 낮춥니다."

## 슬라이드 10 — 중간 산출물 ④ 미션 5카테고리 × 논문 근거 *(2분)*

> *(담당: 반소람·모세종)*

**말할 내용:**

> "미션은 임의로 만든 것이 아닙니다.
> 애도 회복 연구에서 검증된 방법론을 **감정 상태별**로 분류했습니다."

| 카테고리 | 학술 근거 | 미션 예시 |
|---------|----------|---------|
| 기록/정리 | **표현적 글쓰기** — 감정을 글로 꺼내면 마음이 가벼워진다 | 오늘 감정 한 줄 |
| 추모/기억 | **지속적 유대** — 기억 연결이 슬픔을 따뜻함으로 전환 | 사진 한 장 바라보기, 산책길 다시 걷기 |
| 가벼운 활동 | **행동 활성화** — 작은 활동이 가라앉은 기분을 끌어올린다 | 집 앞 5분 산책 |
| 사람과 연결 | **사회적 지지** — 사람과 함께 있으면 슬픔이 덜어진다 | 안부 한 줄, 짧은 통화 |
| 휴식/자기돌봄 | **자기돌봄** — 몸을 쉬게 해야 마음도 회복된다 | 물 한 잔 마시기 |

> "감정 상태에 따라 미션 종류가 달라집니다.
> 많이 힘들 때는 기록·추모·자기돌봄처럼 작은 것부터.
> 조금 나아졌을 때 행동 활성화와 사회적 지지가 들어옵니다.

## 슬라이드 12 — 논문 근거 + 정량 검증 구조 *(5분)*

**말할 내용:**

> "강사님께서 '정량적 입증이 없다'고 하셨습니다. 맞습니다.
>
> 저희가 '우리 앱을 쓰면 회복이 빠르다'고 주장하는 건 아닙니다.
> 현재 단계에서 그 주장은 할 수 없습니다.
>
> 저희가 만든 건 **그 주장을 검증할 수 있는 구조**입니다."

**논문 근거 4개:**

| 기능 | 근거 |
|------|------|
| 위기 감지 임계값 참고 척도 | ICG (Prigerson, 1995) — >25점 = 사회·정신·신체 기능 유의미 손상, α=0.93 (앱 내 risk_level 설계 참고, ICG 진단 대체 아님) |
| 감정 체크인 장기 추이 필요성 | KCI — 펫로스 비애 1년 이상 지속, PTSD 악화 가능 (국내) |
| 추모 영상 — 디지털 교감 선행 사례 | KoreaSCIENCE (2021) — VR 반려동물 교감이 감정을 돕는 디지털 시스템 연구 (탐색적 참고 근거) |
| play_count 가설 | PMC11303913 — 디지털 앱 engagement 변화 = 측정 가능한 행동 마커 |

> "특히 회복 측정 척도 측면에서 말씀드리면,
> 복잡 애도 측정 표준 도구인 **ICG**는 19문항 5점 척도, 0~76점입니다.
> **>25점**이 기능 손상 기준입니다.
> ICG는 저희 앱의 risk_level 설계에 참고한 척도이며,
> **ICG 진단을 대체하는 것이 아닙니다.**
> 저희 앱은 텍스트·감정 점수 기반으로 위험 신호를 조기에 표시하는 내부 지표를 구현했고,
> 향후 실제 ICG 척도와 비교 검증이 가능한 구조입니다.
>
> play_count 감소 패턴 — '처음엔 매일 보다가 점점 덜 보는' 패턴 — 은
> 학계에서 아직 확립된 것은 아닙니다.
> 그래서 저희는 이것을 **가설**로 표현합니다.
> KoreaSCIENCE 국내 연구에서 VR 반려동물 교감이 감정을 돕는 디지털 시스템의 탐색적 선행 사례가 있고,
> PMC11303913에서 engagement 변화가 행동 마커로 활용됐습니다.
> 이 가설을 데이터로 검증할 수 있는 구조를 만들었다는 것이 저희의 주장입니다."

> "정리하면 이렇습니다:

> **관리받으면 빨리 낫는다는 근거는 아직 약하지만,
> 관리와 추적이 필요한 위험군이 존재한다는 근거는 충분합니다.**
>
> 효과를 입증하기보다, **효과를 검증할 수 있는 구조를 만들었습니다.**
>
> 지금까지 구조를 말씀드렸습니다.
> 이제 실제로 보여드리겠습니다."

## 🛡️ 예상 Q&A 대비 *(Q&A 15분)*

**Q1. 정량적 효과를 입증할 수 있나요?**
> "현재는 효과 입증 단계가 아니라 **효과 검증을 위한 데이터 수집 구조 구현 단계**입니다.
> 관리받으면 빨리 낫는다는 근거는 아직 약하지만, 관리와 추적이 필요한 위험군이 존재한다는 근거는 충분합니다.
> ICG는 복잡애도를 수치화하는 대표 척도로, 저희 risk_level 설계의 참고 기준입니다.
> ICG 진단을 대체하는 것이 아니라, 앱 안에서 위험 신호를 조기에 표시하는 내부 지표를 구현했고,
> 향후 ICG 같은 표준화 척도와 앱 로그를 함께 비교해 효과를 검증할 수 있는 구조입니다."

**Q2. RAG를 어디에 어떻게 썼나요?**
> "실제 코드 기준으로 3곳입니다.
> `memorial`(추모 위로 메시지, corpus 45건), `mission`(회복 미션 추천, corpus 5건), `funeral`(장례 안내 Q&A, corpus 13건).
> 각 API 호출 시 `where={"category": ...}` 필터로 컬렉션을 분리합니다.
> 수의사 처치 안내는 RAG 후보였지만 의료 조언 대체 위험 차단을 위해 고정 텍스트로 확정했습니다.
> 미션 추천 시에는 RAG로 예시를 찾고, 행동활성화·사회적 지지 등 논문 기반 회복 근거를 함께 제시합니다."

**Q3. 1인칭 반려동물 화법이 윤리적으로 문제 없나요?**
> "보호자 동의, 경고 문구, 그리고 감정 체크인 전체 구간 risk_level 0 조건 — 세 가지를 모두 충족할 때만 노출합니다.
> 반려동물의 부활이나 사실인 것처럼 주장하지 않으며,
> 'AI가 보호자가 전해준 추억을 바탕으로 재해석한 꿈 속 작별 인사입니다'라는 문구를 항상 표시합니다."

**Q4. 수의사 연계가 실제로 되나요?**
> "현재 MVP는 수의사 추천을 진입 시나리오로 설계했고, 수의사 웹 화면은 MVP 화면만 구현했습니다.
> 실제 병원 연계 기능 연결은 서비스 출시 단계 과제입니다."

**Q5. 경쟁 서비스 대비 차별점이 뭔가요?**
> "시한부 선고 시점 진입, B2B2C 구조, 회복 게이트(위기 보호자에게 추모 콘텐츠 자동 차단),
> 위기 감지 4단계 + 1393 자동 연결 — 이 네 가지가 기존 메모리얼 앱과의 차이입니다."

**Q6. 회복 점수 80점 기준은 어떻게 정했나요?**
> "현재 기준은 팀 내부 합의로 설정했고, 실사용 피드백이 쌓이면 조정할 예정입니다.
> 산식 자체 — 감정 평균 40% + 미션 완료율 35% + 체크인 꾸준함 25% — 는
> ICG 척도에서 회복 추이를 점수 변화로 추적하는 방향성을 참고해 설계했습니다.
> ICG 진단 수치를 그대로 적용한 것은 아닙니다."

**Q7. play_count 감소가 회복이라는 근거가 있나요?**
> "학계에서 확립된 것은 아닙니다.
> 다만 디지털 정신건강 앱 연구에서 engagement 변화가 행동 마커로 활용된 사례 (PMC11303913)와,
> 국내 VR 반려동물 교감 연구 (KoreaSCIENCE, 2021)를 참고해 **검증 가능한 가설**로 설계했습니다.
> 실제 데이터가 쌓이면 이 가설을 검증할 수 있는 구조가 준비돼 있습니다."
