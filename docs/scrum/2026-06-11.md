## 2. 주요 논의

### 2-1. 회복 점수 산식 확정

- 기존: `avg × 10` (단순 평균)
- **신규 확정 산식:**
  ```
  recovery_score = 감정(40%) + 미션누적(35%, sticky 천장 35) + 꾸준함(25%)
  ```
  - 80점 = 콘텐츠 해금 (~12일차)
  - 백엔드 `emotion.py` → `recovery_signal.py` 호출로 교체 (윤한님 구현)

### 2-2. 수면 modifier — 점수 포함 여부 논쟁

- **논의:** 수면 5단계를 회복 점수에 포함할지 여부
- **결정 (a안):** 수면은 점수에 반영 X
  - 이유: 회복을 위한 지표이지, 수면 부족이 회복 의지를 깎아선 안 됨
  - 수면은 **미션 난이도 조정 보조 + 리포트 표시**에만 활용
  - 활동(걸음) 데이터는 optional add-on 10점으로만 유지

### 2-3. 자책감 처리 방식

- **논의:** 자책감 단독 입력 시 위기 레벨 상향 여부
- **결정:**
  - 자책감 단독 → **L0 유지** (공감 톤만, 위기 상향 X)
  - L1 + 자책감 조합 → **L2 상향** (안전망 강화)
  - 이유: 자책감은 펫로스 애도 과정의 자연스러운 감정, 단독으로는 위기 아님

### 2-4. 슈퍼계정 gate 문제

- **문제:** Redis TTL 만료 시 슈퍼계정 gate=locked 반복 → 시연 도중 해금 안 되는 상황
- **해결:** `@rainbow.dev` 도메인 이메일은 코드 레벨에서 항상 `gate=open, score=90` 반환
  - Redis 만료·체크인 유무 무관
  - 슈퍼계정 목록: `super@rainbow.dev` ~ `super5@rainbow.dev`, `bong@rainbow.dev`(모세종 전용)

### 2-5. Expo SDK 52 → 54 업그레이드

- 최신 Expo Go(iOS·Android 공식 버전) 지원 필요 — 팀 피드백 반영
- 민경이님 담당, breaking change(pointerEvents·ImagePicker) 대응 포함
- `newArchEnabled: false` 경고도 함께 제거

### 2-6. 앱 사용 수집 — UsageStatsManager 플랫폼 문제

- **문제:** `UsageStatsManager`는 Android 네이티브 API → Expo Go에서 바로 사용 불가
- **검토한 방식:**
  - A안: 별도 컴패니언 앱으로 JSON 내보내기 → 백엔드 전송
  - **B안: Expo Native Module 래핑** — JS에서 직접 호출 가능하도록 브릿지 작성
- **결정: B안 채택**
  - 기존 작동하는 Android 앱 코드(모세종 직접 작성) 기반으로 래핑
  - `expo-modules-core`로 Kotlin 코드 감싸서 `UsageStats.getUsageStats(date)` JS API 노출
  - Expo Go 대신 **개발 빌드 APK** 필요 (`eas build` 또는 `npx expo run:android`)
  - `PACKAGE_USAGE_STATS` 권한: 사용자가 설정에서 직접 허용 필요 (특수 권한)
- **담당:** 모세종

### 2-7. 프론트 배포 방식 전환 — EAS Build + OTA (논의 중)

- **현재:** `expo start --tunnel` → QR → Expo Go → 개발 PC에서 번들 서빙
- **검토 중인 방식:** `eas build` 최초 1회 → APK/IPA 설치 → 이후 `eas update`로 OTA 자동 반영
  - PC 불필요, QR 불필요, 팀원 앱 재시작 시 최신 반영
- **iOS 옵션:**
  - A안: Apple Developer 계정 없음 → Expo Go + EAS 고정 URL (무료, 일반 배포 불가)
  - B안: TestFlight → Apple Developer 계정 공유 의사 있음 (연락 후 확정 예정)
- **현재 상태:** B안 쪽으로 논의 진행 중, 계정 공유 확인 후 확정 예정
- **담당:** 민경이 (EAS 설정)

### 2-8. LP 좀비 프로세스 문제

- **증상:** 영상 생성 후 python 프로세스가 메모리에 남아 RAM 92% 점유
- **원인:** `conda run` 래퍼가 자식 프로세스를 회수하지 않음
- **해결 (장민수):** `conda run` 제거 → `python.exe` 직접 호출 + `Semaphore(1)` + `Popen.wait()`
  - 소람님 서버 적용·검증 완료: 좀비 0개, RAM 34% (기존 92%)
