## 한 줄 요약

> RAG는 **3곳**에 씁니다 — ① 추모 메시지 · ② 일상복귀 미션 · ③ 장례 안내.
> 검색은 항상 `where={"category": ...}` 로 영역을 나눠서 섞이지 않게 합니다.

## ✅ RAG 쓰는 곳 (3)

| category | 기능 | 쓰는 코드 | corpus | 검색 방향(무엇을 찾나) |
|----------|------|-----------|:---:|------------------------|
| `memorial` | ③ 추모·위로 메시지 생성 | [memorial.py](../ai/llm/memorial.py) | 45건 | 감정 공감·이별 슬픔 |
| `mission` | ⑤ 일상복귀 미션 추천 | [mission.py](../ai/llm/mission.py) | 5건 | 회복 단계별 행동 유도 |
| `funeral` | 장례 안내 Q&A | [funeral.py](../ai/llm/funeral.py) | 13건 | 절차·방법 정보 제공 |

- 동작: 사용자 입력 → 의미가 가까운 corpus 글 top-3 검색 → LLM 프롬프트에 **참고 예시**로 주입 → 생성.
- 미션은 추가로, 추천된 미션마다 **분류별 회복 근거(논문 기반)**가 함께 나갑니다(아래 ※).

## ❌ RAG 안 쓰는 곳 (헷갈리기 쉬움 — 발표 때 명확히)

| 영역 | 왜 RAG 아님 |
|------|-------------|
| **수의사 처치·증상 안내** (`vet_protocol`) | **고정 텍스트로 확정(2026-06-08).** 기본 대처법만 안내하고 "내원해 주세요"로 유도 — 의료 조언 대체 위험 차단 |
| **위기 감지 (1393)** | 안전 시스템에 예측 불가능성(검색 변동)을 넣지 않음 — 의도된 제외 |
| **1인칭 편지** | 윤리 경계(부활·1인칭 금지)와 충돌 → RAG 제외 |

## ⚠️ 가이드 이름 vs 실제 코드 (발표 때 정확히)

발표 메모/팀 가이드([CLAUDE.md](../CLAUDE.md) §3)는 `comfort·mission·funeral·vet_protocol` 4개로 적혀 있으나, **실제 코드와 차이**가 있습니다:

| 가이드 이름 | 실제 상태 |
|-------------|-----------|
| `comfort` | 코드에선 **`memorial`** 로 구현됨 (= 추모·위로 메시지, 같은 기능 / 이름만 다름). ✅ RAG 씀 |
| `mission` | 그대로. ✅ RAG 씀 |
| `funeral` | 그대로. ✅ RAG 씀 |
| `vet_protocol` | RAG 후보였으나 **드랍 → 고정 텍스트**. ❌ RAG 아님 |

> 👉 발표에선 **코드 기준 3곳(추모·미션·장례)**으로 말하는 게 정확합니다.

## ※ "논문 근거"는 어디에? (강사 피드백 대응 포인트)

- 미션 추천 시, 각 미션에 **"왜 회복에 도움 되는지" 한 줄 근거**가 분류별로 붙어 나갑니다.
  → 5개 분류 ↔ 5개 회복 이론 (행동활성화·지속적 유대·표현적 글쓰기·사회적 지지·자기돌봄).
  → 위치: [prompts/mission.py](../ai/llm/prompts/mission.py) `CATEGORY_RATIONALE`.
- 즉 "RAG로 예시를 찾고 + 논문 근거를 함께 제시"하는 구조 → 강사님 "논문이 기능에 안 녹았다" 지적에 대한 답.
