## ④ TTS 음성 낭독

### 개요·목적
③ AI 추모 메시지(텍스트)를 보호자에게 **음성으로 낭독**한다. 메시지 톤과 1:1 매핑되는 TTS 톤을 받아 발화 속도·피치를 조절하고, 합성 실패 시에도 데모가 끊기지 않도록 폴백한다.

### 요구사항
- **기능:** 텍스트 → 음성 파일(mp3) 합성, 톤 4종(warm·calm·hopeful·soft) 선택, 목소리 선택, 긴 텍스트 자동 분할.
- **비기능:** 합성 실패(키 없음·할당량·네트워크) 시 graceful 폴백, 재생 길이(duration) 메타 제공.
- **윤리:** 보호자 대상 위로 낭독 전용. 반려동물 목소리 흉내 금지.

### 인터페이스
```python
# ai/tts/tts.py
synthesize(
    text: str,
    tone: TtsTone = TtsTone.WARM,   # WARM | CALM | HOPEFUL | SOFT
    *, voice: Optional[str] = None,  # AVAILABLE_VOICES 중 하나, None=기본
    filename: Optional[str] = None,
) -> dict
# 반환: {"audio_path": str, "duration": float, "format": "mp3", "fallback": bool}
```

| 톤 | speaking_rate | pitch |
|----|:---:|:---:|
| WARM | 0.92 | -1.0 |
| CALM | 0.95 | 0.0 |
| HOPEFUL | 1.0 | +1.0 |
| SOFT | 0.88 | -2.0 |

**백엔드 API:** `POST /api/v1/tts` (인증 필요, 201) → `TtsCreate` → `services/tts.py generate_tts()` 가 `ai/tts.synthesize()` 를 `asyncio.to_thread` 로 호출, 결과 mp3를 `uploads/tts/` 에 저장하고 `/uploads/tts/{file}` URL 반환(`TtsResponse`).

### 처리 흐름
1. 입력 검증(빈 텍스트 거부) → 톤 파라미터·목소리 이름 해석
2. `_split_text` 로 4500자 단위 문장 분할
3. Google Cloud TTS(`ko-KR-Neural2-A` 기본) 청크별 합성 후 MP3 이어붙임
4. 합성 실패 시 → `_samples/fallback_{tone}.mp3` 로 폴백(`fallback=True`)
5. `ffprobe` 로 실제 재생 길이 측정, 실패 시 글자수 기반 추정

### 엔진 선택 경과 (스파이크)
| 일자 | 활동 | 결론 |
|------|------|------|
| Day2 | Google Cloud TTS 잠정 채택 | 클라우드가 8GB GPU·일정에 안전 |
| Day6 | EL vs Google 비교(STT CER·Gemini 오디오 자동평가) | Google 유지 — Free 키 한국어 confound로 EL 공정평가 불가 |
| Day7 | 사람 블라인드 A/B (EL 한국어 9종 vs Google) | **EL 9전 9승** — 기계음 해소, Day6 결론 뒤집힘. 단 상업=유료 → 채택은 PM 판단 |
| Day8 | Typecast 성우 후보 청취 스파이크 | 후보 4종 압축, 팀 의견 수렴 중 |

- 적용 범위: 폴백 `voiced_url` 한정. **주력은 PERSO/LivePortrait 아바타 낭독 영상**(장민수 영역)이며 TTS는 안정성 폴백 트랙.
- 자동지표(CER·Gemini)는 감정·자연스러움을 못 재 **결정 근거에서 폐기**, 사람 청취로 판정.

### 예외·폴백
- 라이브러리 미설치/인증 실패 → 명확한 `RuntimeError`, 호출부에서 샘플 mp3 폴백.
- `ffprobe` 미설치 → 글자수 추정으로 흡수(합성 자체는 안 막음).

### 테스트·산출물

- 비교 하네스: `compare_elevenlabs.py`·`pairwise_listen.py`·`blind_listen.py`·`analyze_stt.py`·`eval_gemini_audio.py`. 기록: `ENGINE_NOTES.md`·`AZURE_TTS_조사.md`·`TTS_무료유료_비교.md`.

## ⑧ 회복 평가 리포트

### 개요·목적
보호자의 서비스 사용 데이터(LLM 호출 횟수·감정 추이·미션 완료율)를 반려동물별로 집계해 **회복 리포트**로 제공한다(수의사도 조회). 모델 성능이 아니라 **서비스 사용 지표**다.

### 설계 원칙
- 집계 함수 `build_report` 는 **순수 함수** — DB 조회 결과(리스트)를 인자로 받는다. GPU/DB 없이 테스트 가능.
- LLM 로그는 **개인정보 최소화** — 대화 원문(프롬프트/응답)은 저장하지 않고 횟수·종류·지연·성공·위기 등급만 남긴다.

### 인터페이스
```python
# ai/evaluation/report.py
build_report(
    pet_id: str, period: Optional[str] = None,
    *, llm_logs=(), emotion_checkins=(), missions=(),
) -> dict
# 반환: {pet_id, period, usage:{total_calls, by_kind}, emotion_trend, mission_completion_rate, revisit}

# ai/evaluation/logs.py
@dataclass LLMLog(kind, provider, model, pet_id, tokens_in, tokens_out,
                  latency_ms, ok, risk_level, created_at)   # 원문 텍스트 없음
measure_latency()                 # 지연 측정 컨텍스트(예외에도 안전)
log_llm_call(collection, resp, ...)    # 동기(pymongo) 1줄 로깅
await alog_llm_call(collection, ...)   # 비동기(motor) 1줄 로깅, resp=None 허용(라이트 로깅)
```
- `kind`: `message`(③) · `mission`(⑤) · `crisis`(⑦) · `funeral` · `triage`.
- **백엔드 API:** `GET /api/v1/report/{pet_id}?period=` (인증 필요) → `services/report.py get_report()`.

### 처리 흐름
1. (백엔드) `pet_id`·기간으로 `llm_logs`·감정체크인·미션 조회
2. `build_report` 가 받아서 집계: `total_calls`=로그 수, `by_kind`=종류별 카운트, 감정 추이 시간순 정렬, 미션 완료율
3. 프론트 차트/요약 UI로 반환

### 로깅 연결 (Day8, 핸드오프)
- 문제: ⑧ 코어는 완료인데 `llm_logs` 실데이터가 안 흘러 `usage.total_calls=0`(껍데기).
- 방향: **라이트 로깅**(토큰 제외, 횟수·종류·지연·성공만) — `build_report` 가 `len`·`kind` 만 쓰므로 토큰 없이도 리포트가 산다.
- 조치: `alog_llm_call` 추가(backend는 motor/async라 동기 헬퍼 못 씀), backend `message.py`·`mission.py` 호출부를 `try/finally` 로 감싸 best-effort 로깅. ⚠️ 위기 경로 early-return 때문에 `try/finally`+분기별 `kind` 로 KIND_CRISIS 누락 방지.
- ⚠️ 호출부 변경은 모세종 영역 → 사전 공유 후 PR(base dev) 예정.

### 예외·주의
- 로그 저장 실패가 **사용자 응답을 깨면 안 됨** → best-effort `try/except`.
- 이중 writer: 김윤한 `POST /api/v1/llm-logs`(#34) vs 서비스 직접 insert → 쓰기 경로 일원화 협의 필요.
- 지표 의미: "기능 호출 1건=1로그"(LLM API 호출 단위 아님). "LLM 호출 횟수"로 오표기 금지.

### 테스트
- `ai/evaluation/test_logs.py` async 헬퍼 테스트 추가 → `ai/evaluation` **11 passed**.
- 통합 정합 시뮬: 로깅 → `pet_id` 필터 → `build_report` 집계가 `total_calls`·`by_kind` 정확히 맞음 입증(실 DB 없이).

### 남은 일
- 모세종께 backend 로깅 변경 공유 → PR. 토큰 정확 집계(반소람 provider 응답 메타 노출). ② 감정 체크인 경로 위기 로깅 연결.

## RAG 검색 파이프라인

### 개요·목적
추모 메시지·미션·장례·수의사 처치 안내의 근거 문서를 **의미 기반 검색**으로 제공하는 공통 파이프라인. ③ 메시지 생성(반소람)이 import 해 소비한다.

### 인터페이스
```python
# ai/rag/retrieve.py
retrieve(query: str, k=None, *, where=None) -> List[Hit]
# Hit = {"text": str, "score": float(코사인 유사도, 클수록 가까움), "metadata": dict}
# 예: retrieve("산책하던 게 그리워요", k=4, where={"category": "memorial"})
```
- 모듈 구성: `config`(설정·TOP_K) · `embeddings`(쿼리 임베딩) · `ingest`(적재) · `store`(ChromaDB 컬렉션) · `retrieve`(검색).
- 컬렉션은 `category` 메타데이터로 분리: `comfort` · `mission` · `funeral` · `vet_protocol`.

### 처리 흐름
1. 컬렉션 적재(`ingest`) → 2. 쿼리 임베딩(`embed_query`) → 3. ChromaDB top-k 검색(+`where` 필터) → 4. 거리 → 유사도(1-거리) 변환 후 내림차순 반환.

### 검증
- 패러프레이즈 쿼리셋(`ai/rag/eval/eval_queries.json`)으로 Hit@1/MRR 측정 → **12개 전부 rank 1 (Hit@1 100% · MRR 1.0)** = 의미검색 정상.
- 벡터DB 비교·선택 근거: `ai/llm/VECTOR_DB_비교.md`·`VECTOR_DB_심화_스터디.md` (ChromaDB 채택). PR #79.

> 경계: RAG **검색 파이프라인(인프라)** 은 본인이 신설. RAG를 쓰는 **메시지·미션 생성 로직**은 반소람 담당.

## GPU 서빙 인프라

### 개요·목적
RTX 5060(8GB) GPU 서버에서 멀티모달 추론(LivePortrait)을 외부에서 접근 가능하게 상시 호스팅. (LivePortrait **기능 자체는 장민수**, GPU 자원·서빙·터널 운영이 본인 몫.)

### 구성·운영
- 하드웨어: RTX 5060 8GB (드라이버 595.97 / CUDA 13.2).
- 서버: `ai/liveportrait/server.py` 를 백그라운드 호스팅(VS Code 터미널 아님). 재시작 = 프로세스 종료 후 uvicorn 재기동, env 없이 코드 기본값+multiplier만.
- 터널: ngrok **무료 고정 도메인** `rerun-devious-reaffirm.ngrok-free.dev` (`--url` 검증). 로그온 시 서버·터널 자동 기동 등록 → URL 유지.
- GPU 운영 원칙: 8GB 한정 → **온디맨드 로딩**(쓸 때 로드, 끝나면 VRAM 회수). LLM 서버 상시 구동과 동시 사용 빡빡.

### 한계
- RTX 5060(Blackwell sm_120)에서 ctranslate2 4.7.2 GPU inference hang → STT(faster-whisper) 평가는 CPU 사용(CER은 device 무관).
