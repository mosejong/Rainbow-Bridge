# 서비스 방향 — 확장 구조 (2026-06-05)

> 오늘 스크럼 후 논의된 서비스 방향 확장안. 팀 공유용.

---

## 핵심 한 줄

> **"아플 때부터 함께, 떠난 후에도 함께"**
> — 수의사 ↔ 보호자 쌍방향 플랫폼 + 펫로스 케어까지 한 흐름으로 연결

---

## 전체 흐름

```
1단계 — 아플 때 (건강관리)
  보호자: 일기처럼 기록 (밥 1/3 먹음, 기침, 산책 못 함 등)
  수의사: 기록 확인 → 처방/조언 (영양제 추천, 병원 방문 권고 등)
  → 쌍방향 소통 플랫폼

        ↓ 수의사가 자연스럽게 연결

2단계 — 이별 준비 (장례 안내)
  수의사: "이제 준비하실 시간입니다" → 장례 절차 안내 + 서비스 연결
  보호자: 앱 안에서 장례 정보 확인, 업체 연결

        ↓ 동일 앱 내에서 전환

3단계 — 떠난 후 (펫로스 케어)
  1단계에서 쌓인 일기·기록 → 추모 메시지/영상 원천 데이터로 활용
  "봄이가 힘들어도 매일 산책 나가려 했던 거 기억해"
  감정 체크인 → 위기 감지 → 일상 복귀 미션
```

---

## 왜 이 구조가 강한가

| 관점 | 이유 |
|------|------|
| 보호자 | 앱 하나로 아플 때부터 추모까지 끝 — 이탈 없음 |
| 수의사 | 환자 사후관리까지 해주는 프리미엄 서비스로 차별화 |
| 데이터 | 1단계 일기 기록이 3단계 추모 콘텐츠 재료가 됨 (데이터 플라이휠) |
| 마케팅 | 병원이 직접 보호자에게 앱 권유 → 별도 마케팅 비용 최소 |
| RAG | 의료 정보·처방 데이터가 자연스럽게 코퍼스로 쌓임 |

---

## 수익 모델

| 대상 | 수익 방식 |
|------|----------|
| 보호자 | 서비스 이용료 (구독 or 건당) |
| 수의사/병원 | 온라인 진료비 수수료 (B2B) |

---

## 타겟 확장

- 기존: 펫로스를 겪은 보호자
- 확장: **반려동물을 키우는 모든 보호자 + 동물병원**

---

## 클라이언트 플랫폼 확정 (2026-06-05)

| 사용자 | 플랫폼 | 이유 |
|--------|--------|------|
| 수의사·병원 | **웹** | 데스크탑 대시보드, 진료 기록·처방 관리 |
| 보호자(유저) | **앱** (Android, Capacitor) | 일기 기록·감정 체크인·추모 — 모바일이 자연스러움 |

- 프론트 코드는 **하나**(React + Vite), 빌드 타겟만 분리
  - 웹 빌드: `VITE_TARGET=hospital` → 서버 배포
  - 앱 빌드: `VITE_TARGET=user` → Capacitor → Android APK
- iOS 빌드는 Mac 확보 후 추가 검토

---

## 프로토타입 범위와의 관계

지금 구현 중인 **3단계(추모 케어)**가 코어.
1·2단계는 이 구조를 알고 설계하면 나중에 자연스럽게 붙일 수 있음.

> 발표 때: 3단계 완성 시연 + "이 서비스는 1단계부터 연결되는 구조로 확장 예정"으로 포지셔닝.
