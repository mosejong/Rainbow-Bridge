# 확장 구조 역할 분담 (2026-06-05)

> [SERVICE_DIRECTION.md](SERVICE_DIRECTION.md) 확장안(1단계 건강관리 → 2단계 장례안내 → 3단계 펫로스케어) 기준 역할 분담.
> 현재 프로토타입(3단계)은 거의 완료([PROGRESS.md](PROGRESS.md)) → 신규는 **1·2단계 + 데이터 플라이휠**에 무게.
> 작성: 정환주 · 팀 합의 확정 (2026-06-05)

---

## 목표

"아플 때부터 함께, 떠난 후에도 함께" — 한 흐름으로 연결.
1단계 일기 데이터가 3단계 추모 콘텐츠 재료가 되는 **데이터 플라이휠**. 의료 데이터 = **RAG 코퍼스**.

---

## 1단계 — 건강관리 (신규, 핵심 확장)

| 태스크 | 담당 | 비고 |
|------|------|------|
| 일기 기록 데이터모델·API (밥/증상/산책 등) | 모세종 | 기록 스키마 CRUD |
| 수의사 사이드 — 계정 역할분리·처방/조언 쌍방향·진료비 수수료(B2B) 기반 | 김윤한 | 보호자↔수의사 권한 분리 |
| **의료정보 RAG 파이프라인** (코퍼스·임베딩·벡터DB, GPU 호스팅) | **정환주** | 벡터DB 비교 착수 |
| 수의사 조언 생성 보조 (RAG 검색결과 → LLM) | 반소람 | RAG 위에 프롬프트 |
| 일기 입력·수의사 소통 화면 | 민경이 | |

## 2단계 — 이별준비 (신규)

| 태스크 | 담당 | 비고 |
|------|------|------|

## 공통 / 인프라

- 배포·Docker·홈서버: 김윤한·모세종

## 의존성 / 다음 액션

- RAG는 **1단계 일기 데이터모델(모세종) 먼저** → 코퍼스 스키마 확정돼야 임베딩 설계 가능.
- 데이터 플라이휠은 **1·3단계 데이터 연결고리** → 모세종(기록)·반소람(추모 프롬프트)과 3자 합의 필요.
- 다음: 1단계 일기 스키마 초안 확정 → 정환주 RAG 벡터DB 비교안 확정.
