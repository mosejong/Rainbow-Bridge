## §1. 먼저 — 벡터 DB가 하는 일 (그림으로)

우리 서비스에서 실제로 일어나는 일:

```
[1] 보호자가 글 씀:  "산책하던 게 그리워요"
        ↓  (임베딩 모델이 변환)
[2] 숫자 묶음으로:   [0.21, -0.5, 0.8, ... ]   ← 의미를 담은 숫자들
        ↓
[3] 미리 저장해둔 위로글 수천 개의 숫자묶음 중에서
    "숫자가 비슷한 = 의미가 비슷한" 4개를 빠르게 찾음
        ↓
[4] 그 4개를 LLM한테 주고 → 따뜻한 답변 생성
```

이 중 **[3] "비슷한 거 빠르게 찾기"** 를 해주는 창고가 **벡터 DB**입니다. 세 제품(Chroma·Mongo·Pinecone)이 다 이 일을 합니다. 차이는 *어디에 두고, 누가 관리하고, 얼마 드냐*.

### 알아두면 좋은 단어 4개 (쉽게)

- **임베딩(embedding)**
  글자를 "의미가 담긴 숫자 묶음"으로 바꾼 것.
  쉽게 말하면: *문장을 좌표로 바꾸기.* 의미가 비슷한 문장은 좌표도 가까이 찍힘.

- **코사인 유사도**
  두 숫자 묶음이 얼마나 비슷한지 재는 방법(각도로 잼).
  쉽게 말하면: *두 좌표가 같은 방향을 보는가?* 같은 방향이면 "비슷하다".

- **HNSW**
  수천~수백만 개 중에서 비슷한 걸 빨리 찾는 *검색 지름길 지도*. 세 DB 모두 이걸 씁니다.
  쉽게 말하면: 전부 하나하나 비교하면 느리니까, *미리 이웃끼리 길을 이어둔 지도*를 따라 점프하며 빨리 찾는 방식.
  - ⚠️ 이 지도는 **컴퓨터 메모리(RAM)에 올려둬야** 빠릅니다. 그래서 "데이터가 RAM보다 커지면 느려진다"는 한계가 생깁니다(아래 ChromaDB에서 다시 나옴).
  - 손잡이 3개: `M`, `efConstruction`, `efSearch`. 크게 잡으면 *더 정확하지만 더 느리고 메모리 더 먹음*. 작게 잡으면 반대. **정확도 ↔ 속도 맞바꾸기.**

> 한 줄 요약: 벡터 DB = "문장을 좌표로 바꿔 저장해두고, 비슷한 좌표를 빨리 꺼내주는 창고."

## §2. 세 제품 자세히 (비유 + 사실)

### 2-1. ChromaDB — 내 방 책장

**한 줄 비유**: 내 컴퓨터(홈서버) 안에 직접 두는 작은 책장.

**어떻게 생겼나**
- **앱 안에 들어가서 같이 돕니다**(따로 서버 켤 필요 X). 같은 컴퓨터 안이라 *왔다갔다 통신 시간이 거의 0* → 빠름.
  쉽게 말하면: 검색 창고가 *내 프로그램 옆자리*에 붙어 있는 것.
- **2025년에 속을 Rust로 새로 짬** → 예전보다 읽고 쓰기 약 **4배** 빨라짐.
- **데이터는 디스크에 저장**돼서 컴퓨터 꺼도 안 날아갑니다(폴더 하나에 인덱스·기록이 저장됨).

**한계 (솔직하게)**
- 위 §1에서 말한 **검색 지도(HNSW)를 메모리(RAM)에 올려둬야** 함 → **담을 수 있는 양이 RAM 크기에 묶임.** RAM을 넘으면 갑자기 엄청 느려짐.
  쉽게 말하면: *책상 위에 다 못 올리면 바닥(디스크)까지 뒤져야 해서 느려진다.*
- **혼자(컴퓨터 1대) 약 500만~1,000만 개**까지가 실용 한계. 그 위로는 버거움.
- 손잡이가 3개뿐이라 아주 큰 규모에선 세밀 조정이 어려움.

**우리한테**: MVP·개발 단계에 딱. `pip install chromadb` 한 줄, 홈서버에 그냥 얹으면 됨.

```python
from langchain_chroma import Chroma
vectorstore = Chroma(
    collection_name="memorial_msgs",
    embedding_function=embeddings,
    persist_directory="./chroma_db",   # 이 폴더에 저장 (컴퓨터 꺼도 유지)
)
```

### 2-2. MongoDB 벡터검색 — 쓰던 창고에 칸 추가

**한 줄 비유**: 지금 회원·반려동물 정보를 넣어둔 창고(MongoDB) **안에** 벡터 검색 칸을 더 만드는 것.

**어떻게 동작하나**
- MongoDB 본체 옆에 **`mongot`이라는 검색 전용 일꾼**을 하나 더 띄웁니다. 이 일꾼이 원본 데이터가 바뀔 때마다 따라서 검색용 색인을 갱신합니다.
  쉽게 말하면: 창고지기 옆에 *검색 담당 알바(mongot)*를 한 명 더 두는 것.
- 그러면 회원 데이터랑 **같은 창고·같은 명령어**로 벡터 검색이 됩니다 → 창고를 하나로 합칠 수 있음(이게 제일 큰 장점).

**여기가 핵심 — 두 가지 종류가 있음**

| | Atlas (클라우드 버전) | 직접 운영 버전 (Community 8.2+) |
|---|---|---|
| 형태 | 몽고DB 회사가 클라우드로 다 해줌 | 내 서버에 직접 깔아 운영 |
| 완성도 | **정식 출시(GA)** | **미리보기(Public Preview)** — 아직 정식 아님 |
| 되는 환경 | 클라우드 | **리눅스/도커만** (맥·윈도우 ❌) |
| 필요한 버전 | Atlas | **몽고DB 8.2 이상** (8.0은 ❌) |
| `mongot` 알바 | 회사가 관리 | **내가 직접 깔고 운영** |
| 돈 | 클러스터 요금(대략 월 $60대부터) | 추가 비용 없음(운영 수고는 내 몫) |
| 실서비스 가능? | O | ⚠️ 미리보기라 "테스트용", 정식 전엔 비권장 |

> ⚠️ **꼭 알아둘 정정**: 옛날 자료(우리 [VECTOR_DB_비교.md](VECTOR_DB_비교.md) 포함)는 "로컬 MongoDB로는 벡터검색 절대 안 됨"이라고 했는데, **2025년 8.2 버전부터는 내 서버에서도 됩니다**(미리보기 단계).
> 다만 **우리 서버는 아직 7 버전**이고, 미리보기라 불안정 → 그래서 *결론은 그대로 ChromaDB*입니다. (자세히 §5·§7)

**장점**: 창고 하나로 통합(관리·백업 한 번에). **단점**: 직접 하려면 8.2로 올리고 + 알바(mongot) 챙기고 + 아직 미리보기.

### 2-3. Pinecone — 대형 물류센터 임대

**한 줄 비유**: 전문 회사 창고를 빌려 쓰는 것. 내가 관리할 게 거의 없음. 대신 돈 냄.

**어떻게 동작하나**
- 가입 → API 키 받기 → 코드로 그 회사 서버에 저장/검색 요청. 확장·속도·고장 대비 전부 그쪽이 처리.
  쉽게 말하면: *창고 운영을 통째로 외주.*
- **안 쓰면 요금 0**(스케일 투 제로), 쓰면 쓴 만큼(검색·저장량) 과금.
- **내 서버에 깔 수는 없음**(클라우드 전용).

**요금 (2026년, 공식 페이지 기준)**

| 등급 | 내용 |
|---|---|
| 무료(Starter) | 저장 2GB · 인덱스 5개 · 프로젝트 1개. 작은 연습용 |
| 유료(Standard) | **최소 월 $50** + 저장 $0.33/GB·월 + 검색·쓰기 사용량 과금 |

> 참고: 2GB면 대략 35만 개 정도(숫자 묶음 크기에 따라 달라짐).

**우리한테**: 사용자가 폭증해서 수백만~수천만 개가 되면 그때. 지금은 과하고, **데이터가 외부 회사 서버에 저장**된다는 점도 걸림(감정·추모 정보라 민감).

## §3. "세 가지 기능" 비교 — 무엇이 다른가

기능을 항목별로 펼쳐 비교:

| 기능 | ChromaDB | MongoDB | Pinecone |
|------|----------|---------|----------|
| ① 비슷한 거 검색(핵심) | O | O | O |
| ② 조건 걸러 검색(예: 강아지 글만) | O | O (몽고 문법 강력) | O |
| ③ 키워드+의미 같이 검색 | 앱에서 조합 | 풀텍스트+벡터 결합 가능 | 주로 의미 중심 |
| ④ 아주 커졌을 때 확장 | 1대라 한계(~1천만) | 버전·운영 따라 | **자동 무한 확장** |
| ⑤ 저장·백업 | 폴더 파일로 | DB와 한 번에 | 회사가 자동 |
| ⑥ 관리 수고 | 적음 | 중간(알바 챙김) | **거의 없음** |
| ⑦ 데이터 위치 | **내 서버** | 내 서버 or 클라우드 | 외부 클라우드 |
| ⑧ 돈 | 무료 | 직접=무료 / Atlas=유료 | 무료등급+사용량 |

**헷갈리기 쉬운 점 — "DB가 하는 일"과 "앱이 하는 일"은 다름**

"이 DB는 reranking(재정렬) 되나? HyDE는?" 같은 질문이 나올 수 있는데:

> reranking(검색결과 다시 순위매기기), MMR(겹치는 결과 빼고 다양하게), HyDE(가짜 답을 만들어 검색에 활용) 같은 **고급 검색 기술은 DB 기능이 아니라 우리 프로그램(LangChain 같은 RAG 코드) 쪽 기능**입니다.
> 쉽게 말하면: DB는 "비슷한 거 꺼내주기"까지만 하고, 그 결과를 더 똑똑하게 다듬는 건 **우리 코드**가 함. → **어느 DB를 골라도 이 기술들은 똑같이 얹을 수 있음.**

## §4. 왜 *이 세 개만* 비교했나? (다른 후보 뺀 이유)

"Qdrant·Weaviate·pgvector는 왜 안 봤어?"라는 물음에 대한 답:

- **pgvector**: PostgreSQL(다른 종류 DB) 위에서 도는데, **우리는 그게 없음**(우리는 MongoDB). 그것만 쓰려고 새 DB 들일 이유 없음.
- **Qdrant / Weaviate**: 좋은 도구지만 **프로그램을 하나 더 띄워서 운영**해야 함. ChromaDB는 앱에 그냥 내장돼서 훨씬 간단 → MVP엔 Chroma가 편함.
- **Milvus**: 아주 큰 규모용. 설정이 복잡해서 지금 단계엔 과함.
- **FAISS**: 검색 엔진 부품일 뿐이라 *저장·조건검색·수정 기능을 직접 다 만들어야* 함. ChromaDB가 사실 이런 부품 위에 편의기능을 얹어놓은 것.

> 정리: 우리가 고른 셋은 **"제일 쉬운 내 서버용(Chroma) / 쓰던 DB 통합(Mongo) / 통째 외주(Pinecone)"** 라는 대표 3가지 길. 나머지는 이 셋의 변형이라 대표로 추렸음.

## §5. 우리 프로젝트엔 뭐가 맞나 (결론)

**먼저, 우리 상황 사실 정리** (코드·문서로 확인함)

| 항목 | 우리 상황 |
|------|-----------|
| 지금 MongoDB | 내 서버에 직접 운영하는 **7 버전**(아직 8.2 아님) |
| 임베딩 모델 | **아직 안 정함** |
| 데이터 양 | 미정(예시로 "수천 개") |
| GPU | RTX 5060, 메모리 **8GB**(작은 편) |
| 개발용 LLM | Gemini API로 정함(로컬 폴백 EXAONE) |
| 서비스 성격 | 한국어 펫로스, **민감한 감정·추모 데이터** |

### 결론: **지금은 ChromaDB ★**

이유를 쉽게:
- 💰 **공짜 + 5분이면 시작** → 초보 팀·빠른 MVP에 딱.
- 🔒 **데이터가 내 서버에만 남음** → 감정·추모는 민감 정보라 외부 클라우드로 안 나가는 게 안전.
- 📦 **홈서버에 그냥 얹기 쉬움**(따로 서버 운영 X).
- 📏 **수천~수만 개는 1대로 충분** → Chroma 한계(~1천만)에 한참 못 미침.

### MongoDB로 합치는 건 — *지금은 아직* (나중엔 매력적)
- "창고 하나로 통합"은 분명 좋음. 그런데 지금 하려면:
  1. 서버를 **7 → 8.2로 올려야** 하고(운영 위험),
  2. 그 기능이 아직 **미리보기(정식 아님)**,
  3. **mongot 알바**까지 챙겨야 함.
- → **정식(GA) 나오고 안정되면 그때 다시 검토.**

### Pinecone — 지금은 과함
- 규모가 작아서 오버스펙 + **데이터가 외부에 저장**(민감 정보라 부담). 사용자 폭증하면 그때.

### 임베딩 모델 — 제안만 (아직 결정 X)
> ⚠️ 엔진·임베딩 **최종 결정은 AI 담당(반소람·정환주) 협의** 사항. 아래는 *후보 제안*이지 확정 아님. `.env`에 안 채움.

- 한국어 서비스라 → **BGE-M3**(한국어·다국어 잘함, 숫자묶음 크기 1024, 내 서버에서 무료로 돌릴 수 있음)가 1순위 후보.
- 대안: `multilingual-e5-large`(1024), 한국어 특화 `bge-m3-ko`. API로 가면 OpenAI `text-embedding-3-large`(3072).
- **임베딩을 먼저 정해야** DB 칸 크기(차원수)를 맞출 수 있음 → 임베딩 결정이 먼저.
- GPU 8GB라 빠듯하지만, LLM은 Gemini(클라우드)라 GPU가 비어서 BGE-M3(1GB 안팎) 정도는 여유.

## §6. 자주 나오는 질문 (FAQ)

**Q1. 왜 Pinecone 같은 전문 DB 안 써?**
A. 지금 규모(수천~수만)엔 너무 큼이고, **민감한 추모·감정 데이터가 외부 서버에 저장**돼서 부담입니다. Chroma는 공짜·내 서버 보관·5분 시작이라 MVP에 맞아요. 사용자 폭증하면 그때 Pinecone 봅니다.

**Q2. 로컬(내 서버) MongoDB로 벡터검색 정말 안 돼?**
A. **됩니다 — 조건부로.** 8.2 버전부터 내 서버에서도 되지만 아직 **미리보기**이고, 우리 서버는 **7 버전**이라 올려야 합니다. 그 수고·위험 때문에 지금은 Chroma가 낫습니다. (옛 자료의 "절대 안 됨"은 틀렸습니다 → §7)

**Q3. ChromaDB는 한계가 뭐야?**
A. 검색 지도를 **메모리에 올려야** 해서 담을 양이 RAM에 묶이고, **1대로 약 500만~1천만 개**가 한계입니다. 우리 규모는 한참 아래라 괜찮지만, 아주 커지면 다른 DB로 옮겨야 합니다.

**Q4. HNSW가 뭐야? 왜 빨라?**
A. 비슷한 것끼리 미리 길을 이어둔 *검색 지도*예요. 전부 하나씩 비교 안 하고 지도를 따라 점프해서 빨리 찾습니다. 손잡이(efSearch)로 정확도와 속도를 조절합니다.

**Q5. 임베딩 숫자묶음 크기(차원)가 DB랑 무슨 상관?**
A. DB 칸을 만들 때 **크기를 미리 고정**해야 하고, 크면 저장·메모리·검색 비용이 늘어요. 그래서 **임베딩 모델 먼저 정하고** DB 칸 크기를 맞춥니다(예: BGE-M3=1024).

**Q6. 키워드 검색 + 의미 검색 같이 돼?**
A. MongoDB는 자체로 둘을 합칠 수 있고, Chroma·Pinecone는 **우리 코드에서 둘을 섞어** 씁니다. 어느 DB든 가능합니다.

**Q7. reranking·HyDE 같은 고급 기술은 DB가 해줘?**
A. 아니요. 그건 **우리 프로그램(RAG 코드) 쪽 기능**입니다. DB 선택과 상관없이 똑같이 얹을 수 있어요.

**Q8. 왜 데이터가 민감하면 DB 선택이 달라져?**
A. 펫로스는 사별·우울 같은 민감 정보라 **내 서버 보관(Chroma)** 이 외부 클라우드(Pinecone/Atlas)보다 안전합니다. 이게 우리 선택의 큰 이유 중 하나예요.

**Q9. 데이터가 수십만~수백만으로 커지면?**
A. Chroma는 1천만 부근에서 버거워져요. 그때 (a) MongoDB 8.2 정식으로 통합하거나 (b) Pinecone/Qdrant로 옮깁니다. **저장 형식을 표준으로 맞춰두면** 옮기는 비용이 작습니다.

**Q10. Qdrant·Weaviate·pgvector는 왜 뺐어?**
A. pgvector는 우리한테 없는 Postgres가 필요하고, Qdrant·Weaviate는 **프로그램을 하나 더 운영**해야 해서 MVP엔 과합니다. 우리 후보 셋이 이미 대표 3가지 길을 커버합니다.
